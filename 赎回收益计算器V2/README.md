# ETF 赎回收益计算器

## 用途

读取完整的 QMT1、QMT2、QMT3 交割单和 IB 活动报表，自动计算 `159518` 每个赎回篮子的国内收益、XOP 对冲收益及人民币合计收益。

程序独立于 `V15-ETF`，只复用了其 PyQt5 界面风格和刷新结构。原有 LOF 赎回日期模块未参与本程序计算。

界面字体按操作系统自动选择：Windows 优先使用 `Microsoft YaHei UI` / `Microsoft YaHei` 并启用 Qt 高 DPI 缩放，macOS 使用 `PingFang SC`，Linux 优先使用 Noto CJK，避免跨电脑时回退到宋体或出现高分屏字体模糊。

## 当前规则

1. QMT1、QMT2 被视为一个统一虚拟账户。
2. QMT3 是独立业务系统，只计入 `159518` 的库存成本、场内自平和移仓成本；QMT3 永不生成赎回篮子，也不会读取退款或现金差额。
3. QMT3 只有在同交易日、`QMT3 卖出 -> QMT1/QMT2 买入`、数量完全相同且成交均价差不超过 `0.002` 元时，才识别为移仓。移仓后的赎回成本为 `QMT3 原FIFO成本 + 接收账户买入实际成本 - QMT3 卖出净额`，因此包含两套系统的价差和费用；不符合条件的 QMT3 卖出保留为普通场内自平。
4. QMT1/QMT2 记录按 `交收日期 + 合同编号 + 来源账户 + 源行号` 排序；QMT3 使用其 `成交日期 + 成交时间 + 合同编号`，并保留字母数字合同号。
5. `证券买入` 进入统一 FIFO 库存，成本取 `发生金额` 绝对值，因此包含交割单中的实际费用。
6. `证券卖出` 优先从 FIFO 库存剥离并单独计算场内平仓收益，不进入赎回篮子；若共享文件夹中存在同日 `QMT成交时间.csv`，程序会用真实成交处理时间辅助匹配该场内平仓对应的 IB 回补。
7. `ETF 基金赎回` 消耗剥离后的剩余 FIFO 库存。
8. `ETF 现金差额` 和 `ETF 申购退款` 先按 `来源账户 + 合同编号` 找候选篮子，再按该流水日期与候选篮子的预计 T+3 / T+6 日期距离归入最接近的篮子，避免不同交易日重复合同编号导致现金流重复挂账。
9. 篮子汇总、篮子配对、IB 成交分配和盈亏归因始终按每 `1,000,000` 份 `159518` 对应 `990` 股 XOP 计算；PCF 校准和手动股数不会代入主账务链路。
10. IB 佣金按实际成交记录计入；若只使用一笔成交的一部分，佣金和成交额按数量比例分摊。
11. IB 会先解析全部股票成交。若某一时间窗口内出现大量 XOP 成分券同步卖出，附近的 XOP 买入先作为底仓处理；后续卖出该底仓时，按卖出 XOP 价格生成 `synthetic_open` 合成 XOP 空头。没有成分券窗口支持的 XOP 买入仍按普通直接空头回补处理。
12. IB 借券费按费用起息日仍处于未平仓状态的直接 XOP 空头股数比例分摊；合成 XOP 空头暂按 XOP 等价价格估值，不逐只成分券计价。
13. 汇率由界面统一手动设置。人工指定某篮子 IB 成交或人工标记虚拟平仓后，覆盖记录保存在 `ib_mapping_overrides.json`。
14. 程序自动识别 `ETF 基金赎回`，按赎回日计算预计 T+3 现金差额日和 T+6 现金替代款到账日；T+3 现金差额只跳过周末，T+6 现金替代款会额外跳过“休市日”中人工维护的日期。
15. 同日、不同 QMT 账户、方向相反、数量完全相同且合同号间隔不超过设置值的成交，自动识别为跨账户调仓；支持先卖后买和先买后卖。若同日跨账户买卖聚合总量完全相等且至少一侧多笔，程序会作为聚合调仓兜底识别。程序仍按合同号顺序执行，卖出消耗当时的旧 FIFO，买入形成新成本，调仓卖出不会进入普通场内平仓路径。
16. 若某个篮子没有实际 XOP BUY 平仓，而是次交易日买入 `159518` 承接风险敞口，可在篮子明细中勾选“该篮子使用次交易日国内买入作为虚拟平仓”。程序会用次交易日未被场内平仓/调仓消耗的国内买入折算 XOP 等价价格，生成 `domestic_rollover_close` 和 `domestic_rollover_open`：前者用于旧篮子平仓，后者沿对应 `159518` 买入的国内 FIFO 成本溯源，归入最终实际消耗该批份额的后续赎回篮子，而不是简单归入紧邻的下一篮子或重新落回全局 IB FIFO。
17. “国内外碎单自平”以国内普通场内卖出为业务起点：国内卖出按 FIFO 计算净卖出额、对应持仓成本和国内盈亏，再配对等价数量的 XOP 做空开仓与买入平仓，合并计算国内人民币盈亏、IB 美元盈亏及折算后的策略总盈亏。开仓允许来自前一交易日；平仓默认使用国内卖出当日中国交易时段的 IB 买入，也可人工指定跨日平仓。
18. 篮子、国内场内自平、QMT3 承接和滚动头寸共享同一套按数量切片的分配账本。每一股 IB 成交和每一份国内买入最多只能归属一个业务切片；参与赎回的数量不能再次进入碎单自平。计算完成后会执行唯一性审计，发现超量或重复归因时直接停止并报错。
19. 所有业务分配完成后的剩余 IB 成交仍按时间 FIFO 配对，但仅作为“未归因 IB 残余诊断”展示，不计入国内外碎单自平收益；仍无法闭合的分片保留为未闭合风险头寸。

## 到账估算与校准

主界面的“到账预估与校准工具”内部拆分为四个独立功能页。每页顶部都显示用途、操作步骤和公式，计算结果按“完整字段名称 / 计算结果 / 数据来源或公式”纵向展示。

### 1. 指定日期到账预估

用于预估没有出现在 QMT 中的假设赎回。

1. 选择国内发起赎回的交易日。
2. 输入假设赎回份额，通常为 `1,000,000` 份。
3. 默认不勾选“手动指定每申赎单位XOP股数”，程序使用 PCF 扣除现金差额后的反推股数。如需做 `998` 股等手工假设，勾选该选项并输入股数。
4. 点击“计算指定日期的预估到账金额”。

手动指定股数只覆盖当前这一次到账预估，不会写入 `pcf_calibration_points.csv`，也不会改变篮子汇总、篮子配对、IB 成交分配和盈亏归因所使用的固定 `990` 股。

结果包含 PCF 的资产估值日、SAFE 中间价、XOP 估值日收盘价、两种 PCF 反推股数、本次采用股数、XOP 卖出价格、CFETS 收盘价、预估 ETF 申购退款、预估 ETF 现金差额和预估国内总回款。

```text
预估XOP股数 = 采用的每申赎单位XOP股数 × 赎回份额 ÷ 最小申赎单位
预估ETF申购退款 = 预估XOP股数 × XOP卖出价格 × 赎回日CFETS美元人民币收盘价
预估ETF现金差额 = PCF EstimateCashComponent × 赎回份额比例
预估国内总回款 = 预估ETF申购退款 + 预估ETF现金差额
```

### 2. 当前 QMT 篮子预估

自动对当前 QMT 交割单中已发起的每个赎回篮子进行同样的预估。QMT 已有“ETF 现金差额”时优先使用实际值；已有“ETF 申购退款”时，同时显示实际值和预估值。

### 3. PCF 隐含 XOP 股数校准

1. 选择 PCF 交易日。
2. 点击“计算并保存该日 PCF 股数校准”。
3. 在“本次校准计算明细”中查看每个输入、两种反推股数和置信度。

```text
按资产净值反推的股数 = NAVperCU ÷ PCF估值日SAFE中间价 ÷ XOP估值日收盘价
扣除现金差额后反推的股数 = (NAVperCU - CashComponent) ÷ SAFE中间价 ÷ XOP收盘价
```

程序默认采用“扣除现金差额后反推的股数”。保存后，该校准点只供指定日期预估、QMT 篮子预估和实际到账反校准使用，不会改变主账务链路固定的 `990` 股。

### 4. 实际到账反校准

用于校验别人或本账户的实际赎回到账数据。

1. 输入赎回交易日和实际赎回份额。
2. “交割单中的 ETF 申购退款”只填补券退款，不能填补券退款与现金差额之和。
3. “交割单中的 ETF 现金差额”单独填写，该值只用于计算实际总回款，不参与 XOP 股数反推。
4. 点击“用外部实际到账数据反推 XOP 股数”。

```text
实际到账反推股数 = 实际ETF申购退款 ÷ 赎回日CFETS收盘价 ÷ XOP卖出价格代理 ÷ (赎回份额 ÷ 最小申赎单位)
反推误差 = 实际到账反推股数 - PCF扣除现金差额后反推股数
实际国内总回款 = 实际ETF申购退款 + 实际ETF现金差额
```

### 数据来源

- `market_data/xop_prices.csv` 保存 XOP 日收盘、尾盘 VWAP、15:59 一分钟收盘价和 16:00 价格。
- PCF 反推只使用 SAFE 人民币中间价；到账预估和实际到账反推只使用 CFETS 收盘参考汇率，两种口径不混用。
- 本地缺少 XOP 价格时，界面通过 TWS `127.0.0.1:7496` / client ID `8888` 只读回填。
- PCF 校准点和历史到账反校准分别保存在 `calibration/pcf_calibration_points.csv` 和 `calibration/settlement_observations.csv`。

## 实时溢价率

“实时溢价率”页面与历史篮子账务相互独立，按“证券资产 + 现金差额”的总篮子资产估算一个 `1,000,000` 份篮子的实时价值：

```text
Bid篮子总资产 = 996 × XOP Bid × CFETS最新美元人民币参考价 + 当日PCF EstimateCashComponent
Ask篮子总资产 = 996 × XOP Ask × CFETS最新美元人民币参考价 + 当日PCF EstimateCashComponent
Bid/Ask每份预估净值 = 对应篮子估值 ÷ 1,000,000
同侧买一溢折价率 = 159518买一 ÷ Bid预估净值 - 1
同侧卖一溢折价率 = 159518卖一 ÷ Ask预估净值 - 1
```

- XOP Bid/Ask 来自 IB TWS 流式实时行情，合约为 `conId 413951498 / STK / SMART / ARCA / USD`，可持续接收盘前、盘后和夜盘数据。
- `159518` 买一/卖一来自新浪 `sz159518` L1 接口，每 3 秒刷新。
- 当日 PCF 必须已缓存且包含 `EstimateCashComponent`；缺失时页面停止总资产估值，不会把现金差额默认为零。程序启动后会由“申购赎回清单”页在后台预取。
- 页面按“卖5→卖1、买1→买5”显示新浪上下五档；每一档同时计算相对总资产 Bid 净值和总资产 Ask 净值的两套溢折价率。
- 汇率来自中国外汇交易中心当天最新可用小时参考价，每分钟检查一次。当天尚未发布小时价时回退最近一个有数据的交易日；页面和 JSON 都显示实际交易日及报价时点。
- 页面提供 IB 连接和断开按钮，默认连接 `127.0.0.1:7496` 并按当前进程自动分配 client ID；若 TWS 返回 326 占号错误，会立即切换备用 ID。可在“数据源设置”中关闭自动分配、指定手工首选 ID；手工首选发生冲突时仍会自动换号。
- 该页面固定 `996` 股只用于实时总资产估值；篮子汇总、篮子配对、IB 分配和历史盈亏仍固定使用 `990` 股，条件单默认数量也未自动更改。
- 配置项 `shared_folder_path` 指定共享根目录。四项实时数据（XOP、159518、CFETS、PCF）完整后，每次更新都会原子覆盖写入 `共享根目录/YYYYMMDD/realtime.json`，并在同目录维护 `realtime字段说明.md`。JSON schema 已升为 2，包含 PCF 现金差额、证券资产和总篮子资产；盘口仍只输出买二、买一、卖一、卖二的价格和溢折价率，不包含盘口数量。

## XOP 晚间平仓条件单

“XOP晚间平仓条件单”页复用实时溢价率页面的 TWS 连接。单篮子生成5张预览，默认990股分配为200、200、200、200、190，触发时间为美东15:58:45、:52、15:59:00、:07、:15。两篮子生成10张预览，默认1980股分配为9张200股和最后1张180股，触发时间从15:58:45均匀铺至15:59:15。所有触发时间强制位于15:58:45至15:59:15，最后一笔距离16:00常规收盘45秒，保留补单缓冲。

每张订单固定为 `XOP / BUY / MKT / DAY / SMART / primaryExchange=ARCA`，使用 `TimeCondition(isMore=true)`，并固定 `outsideRth=false`、`conditionsIgnoreRth=false`、`conditionsCancelOrder=false`、`transmit=true`。

防误触流程：

1. 生成预览不会发送订单，所有订单默认不勾选。
2. 只处理用户逐行勾选的订单。
3. 必须另外勾选“解锁实盘下单”。
4. 每一张订单均弹出独立确认框，默认按钮为“否”；只有明确选择“是”才进入TWS发送队列。
5. 周末日期、已过触发时间、数量合计错误、最后一单不大于0以及重复 `orderRef` 均禁止发送。
6. 页面监听并显示 `openOrder`、`orderStatus`、`execDetails` 和 `commissionReport` 回报。

TWS 只读回填命令（可选）：

```bash
conda run -n ag python backfill_xop_from_tws.py \
  --start 2026-06-18 --end 2026-07-02 \
  --intraday-days 2026-06-22,2026-06-23,2026-06-25,2026-06-29,2026-06-30 \
  --host 127.0.0.1 --port 7496 --client-id 8888
```

## 篮子配对图

“篮子配对图”用三栏同步展示匹配结果：左侧为 QMT1/QMT2 合并后的 `159518` 成交，中间为赎回篮子，右侧为 IB `XOP` 成交。

- 国内成交先按日期混合 QMT1/QMT2，再按合同编号、账户和源行号排序；`Seq` 是每天从 1 开始生成的合并日内顺序。
- 同一笔国内或 IB 成交若被拆给多个篮子，会按实际采用数量拆成多行，避免一笔成交只能显示一种篮子颜色。
- 篮子使用有限组浅色循环显示；左右成交与中间篮子同色，虚线连接各篮子在当前可见区域内的首尾成交。
- 灰色和同栏虚线括号表示国内自买自平，橙色表示 QMT1/QMT2 跨账户调仓，紫色表示未进入任何业务分配的 IB 残余 FIFO 配对，白色表示配对后仍未闭合的风险头寸。
- 三栏可独立上下滚动，透明绘图层会在滚动时动态重画连接线。顶部日期区间会分别按 QMT 成交日、赎回日和 IB 账单时间过滤。

## 国内外碎单自平

“国内外碎单自平”页面只统计与国内 `159518` 普通场内卖出相对应的完整套利闭环，不再把单纯的 IB 剩余成交 FIFO 配对当成业务自平收益。

- 每一行代表一笔国内场内卖出策略，展示国内卖出份额、等价 XOP 数量、IB 开仓/平仓股数、国内 FIFO 成本、净卖出额、国内盈亏、IB 开平均价、佣金、IB 盈亏和合并后的人民币总盈亏。
- 顶部卡片分别汇总已完整闭合的策略数、国内盈亏、IB USD/RMB 盈亏和策略总盈亏；IB 开仓或平仓不足的行仅作为待处理策略显示，不进入已实现汇总。
- 国内卖出使用与赎回相同的 FIFO 库存剥离结果；IB 开仓和平仓使用统一按数量切片账本。即使同一原始成交被拆分给碎单和篮子，两者采用数量也互不重叠。
- 自动平仓只认国内卖出日北京时间 `09:00` 以后的中国交易时段 IB 买入。若实际平仓跨天，可选中策略后人工指定 IB 开仓/平仓成交；人工映射会持久化到 `ib_mapping_overrides.json`。
- 下方“国内 FIFO 来源”“采用的 IB 开仓”“采用的 IB 平仓”用于逐笔审计。未归因 IB 残余 FIFO 配对和未闭合风险头寸放在独立诊断页中，不参与策略收益合计。
- 导出篮子 CSV 时，若存在碎单策略，会同时生成同目录、同文件名前缀的 `_碎单自平.csv` 审计文件。

篮子汇总中的“预估总收益”使用当前篮子已经归属的 IB 净盈亏，并以预估篮子资产替代尚未到账的国内实际回款：

```text
预估总收益 = 预估篮子资产 - 国内成本 + IB净盈亏RMB
篮子偏差 = 篮子资产 / 预估篮子资产 - 1
收益偏差 = 实际合计RMB - 预估总收益
```

没有有效预估篮子资产时显示 `--`；实际退款或现金差额尚未齐备时，“篮子偏差”和“收益偏差”显示 `--`。IB股数、IB净盈亏USD及IB净盈亏RMB只在“显示详细”模式展示。这些字段不改变篮子结算状态、实际收益或行颜色。

## 结算状态

- `待现金差额`：未发现归属于该篮子的 `ETF 现金差额`。
- `待现金替代款`：已有现金差额，但尚未发现 `ETF 申购退款`。
- `IB未完全匹配`：990 股开仓或平仓不足。
- `国内库存不足`：统一 FIFO 库存不足以覆盖赎回份额。
- `已结算`：国内库存、IB 开平仓、现金差额和现金替代退款均完整。

未结算篮子仍显示暂算金额，但只有 `已结算` 篮子会进入顶部“已结算收益”合计。

## 到账日期与休市日

- 实际现金差额日取归属于该篮子的 `ETF 现金差额` 流水日期。
- 实际到账日取归属于该篮子的最后一笔 `ETF 申购退款` 流水日期。
- 预计现金差额日按赎回日后第 3 个境内工作日计算，只自动跳过周六、周日。
- 预计现金替代款到账日按赎回日后第 6 个工作日计算，会跳过周末和手工维护的休市日。
- 点击顶部“休市日”，可从日历添加或删除节假日；修改后 T+6 预计日期会立即重算，实际流水日期和收益不受影响。
- 篮子汇总默认只显示预计 `T+3`、预计 `T+6到账`，并隐藏合同编号；点击顶部“显示详细”后展开预计/实际日期和合同编号。
- 在“篮子汇总”中双击任意篮子，可打开二级明细窗口，分别查看该篮子采用的国内 FIFO 买入、IB 做空开仓和 IB 回补平仓记录；同一窗口可勾选并保存该篮子的人工虚拟平仓标记。

完整 QMT1/QMT2 交割单没有成交时间，因此其跨账户调仓使用合同编号顺序作为时间代理。若“共享文件夹目录路径”下存在 `YYYYMMDD/QMT成交时间.csv`，程序会用 `来源 + 委托号/合同号后缀 + 日期 + 方向 + 数量` 给 QMT 记录补真实成交处理时间，例如交割单合同号 `3800029805` 会匹配委托号 `29805`。可在“数据源设置”调整“调仓合同号最大间隔”。QMT3 使用自身的成交时间和字母数字合同编号，跨系统移仓不使用合同号间隔，而采用同日、精确数量和价格容差的保守规则。

## 批量补全近两个月 PCF

`backfill_pcf_two_months.py` 会扫描已纳入列表的 SZSE/SSE ETF，在默认最近两个自然月内只补本地缺失的 XML PCF。两家交易所各自一条串行队列、彼此并行；同一交易所相邻 HTTP 请求至少间隔 5 秒。可中断后直接重跑，已缓存 XML 会自动跳过。

```bash
cd /Users/ellis/Desktop/ETF交割/赎回收益计算器V2
conda run -n ag python backfill_pcf_two_months.py
```

只查看待补数量而不联网：

```bash
conda run -n ag python backfill_pcf_two_months.py --dry-run
```

脚本仅批量下载结构化 XML，不下载可选原始 TXT，以避免深交所批量任务把请求数翻倍。运行前建议关闭主界面的 PCF 自动预取；若交易所返回限流，脚本会遵守冷却期退出，稍后重跑即可续传。

## 本次实现经验

1. 可视化不能从汇总金额反推关系，计算引擎必须同时输出原始 QMT 成交、篮子 FIFO 分片、场内平仓分片、调仓 FIFO 分片和 IB 分片，UI 只消费这些可审计结果。
2. 一笔成交可能被多个篮子或特殊路径部分使用，因此颜色应绑定“采用数量分片”，不能直接绑定原始成交行。
3. QMT1/QMT2 调仓必须在普通场内卖出之前走独立路径，并同时支持先买后卖、先卖后买；目前使用同日、跨账户、反向、数量完全一致和合同号邻近五个保守条件。
4. Qt 可以完成滚动列表上的动态连线，不需要嵌入 Web 页面。透明覆盖层与表格是兄弟控件，坐标必须经全局坐标转换，不能把兄弟控件直接作为 `mapTo` 目标，否则会进入未定义行为。
5. 国内外碎单自平必须直接消费计算引擎产出的场内卖出 FIFO 分片，并与篮子、QMT3、滚动头寸共享统一的 IB 数量容量；不能由 UI 重新推算。业务分配后的 IB 残余 FIFO 配对仅作诊断，仍无法闭合的数量保留为未闭合风险头寸。
6. QMT 合同编号可能跨交易日重复，现金差额和退款不能只按合同号挂账，必须结合赎回日和预计到账窗口做二次分配。
7. IB 的 `O/C` 标记只能作为分片线索，不能简单等同于策略意图。程序需要先按 XOP 持仓拆分同一笔混合成交，再结合成分券同步做空窗口判断哪些 XOP 买入是底仓、哪些是普通回补。
8. 合成 XOP 空头按 XOP 等价价格计价：买入底仓本身不进普通回补池，卖出底仓形成 `synthetic_open`，用于篮子开仓价格；目前不逐只成分券计价。
9. 美股休市日不能直接套到所有国内流程：`2026-07-03` 这类美国休市日会影响 T+6 退款推算，但不应把境内 T+3 现金差额从 `2026-07-03` 推到 `2026-07-06`。
10. 篮子 IB 平仓不能简单吃最早 BUY。程序现在先把北京时间中国交易时段内、与国内 `证券卖出` 对应的 XOP BUY 剥离为“国内自平 IB 回补”，再让篮子从赎回日美东 `15:00` 后开始匹配回补 BUY，避免第 2/4/5 篮子错吃前序日内自平。
11. 成分券做空造成的 XOP 底仓识别必须限制在成分券同步交易窗口附近；不能让历史 component gap 无限期吞掉后续 XOP BUY，否则 7/7 这类正常回补会被错误当作底仓。
12. 虚拟平仓需要可持久化到单个篮子，因为它是交易意图层面的口径，不一定能仅凭流水自动稳定判断。当前入口放在篮子明细页，勾选后会清除该篮子的人工 close 选择，避免真实 XOP BUY 和虚拟平仓双重匹配。
13. 国内滚动套利的虚拟开仓必须跟随对应 `159518` 买入批次的国内 FIFO 去向；次日仍可能先赎回旧库存，因此不能假设虚拟开仓一定属于日期紧邻的下一个篮子。
