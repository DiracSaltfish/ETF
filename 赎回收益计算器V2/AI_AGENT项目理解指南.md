## 1. 项目一句话定位

这是一个面向 `159518` 的 PyQt5 桌面账务工具：读取 QMT1/QMT2/QMT3 完整交割单和 IB 活动报表，将国内 ETF 库存、场内卖出、赎回现金流、XOP 对冲成交及借券费拆分到可审计的赎回篮子，计算每篮子的国内收益、IB 收益和人民币合计收益。

项目同时包含三类相互隔离的辅助能力：

1. PCF、汇率、XOP 历史价格的拉取和本地缓存；
2. 到账预测、PCF 隐含 XOP 股数校准、实际到账反校准；
3. 实时溢价率、共享 JSON 输出和 XOP 晚间平仓条件单。

`159605` 相关文件目前是研究脚本和方案文档，不进入 `159518` 正式主账务链路。

## 2. 最重要的三套数量口径

这是后续修改时最容易混淆、也最不能随意统一的规则。

| 使用场景 | 每 1,000,000 份对应 XOP | 说明 |
|---|---:|---|
| 历史主账、篮子配对、IB 分配、场内自平、晚间条件单 | `990` 股 | `redemption_engine.DEFAULT_HEDGE_SHARES` 和 `xop_close_orders.DEFAULT_TOTAL_QTY`；当前 UI 主计算不传动态目标 |
| 篮子资产预测、实时总资产估值 | `996` 股 | `settlement_estimator.PREDICTED_REFUND_SHARES_PER_CU` 和 `realtime_premium.DEFAULT_XOP_SHARES` |
| PCF 校准、指定日期到账估算 | 动态 `q_net`，也可单次手工覆盖 | 仅用于预测/校准，不回写历史主账 |

结论：除非需求明确要求迁移账务口径并重新验证历史归因，否则不要把 `990`、`996` 和动态 `q_net` 合并为一个全局常量。

## 3. 主数据流

```text
config.json
  ├─ QMT1/QMT2/QMT3 Excel
  ├─ IB Activity CSV
  ├─ 人工汇率、休市日、调仓合同号间隔
  └─ PCF/汇率/XOP/校准缓存路径
          │
          ▼
redemption_ui.MainWindow.calculate
          │
          ▼
redemption_engine.calculate
  1. 读取、规范化并排序 QMT
  2. 识别 QMT1/QMT2 调仓和 QMT3 成本承接
  3. 建立国内 FIFO：买入 → 调仓/场内卖出 → 赎回篮子
  4. 按合同和预计到账窗口挂接现金差额/退款
  5. 应用人工退款和对冲目标
  6. 读取 IB XOP、全部股票成交和借券费
  7. 构造 direct/synthetic XOP 等价成交流
  8. 先分配国内场内自平，再分配赎回篮子
  9. 必要时用次交易日国内买入生成虚拟平仓/再开仓
 10. 分摊借券费、计算状态和收益
          │
          ▼
CalculationResult
  ├─ baskets
  ├─ venue_closes
  ├─ account_transfers
  ├─ qmt_records / ib_trades / borrow_fees
  └─ 未分配 IB 数量、全局警告
```

## 5. 国内端匹配原则与优先级

### 5.1 输入过滤和规范化

1. 只保留证券代码规范化后为 `159518` 的记录。
2. QMT1/QMT2 保留原操作名称；QMT3 只接受：
   - `买入` → `证券买入`
   - `卖出` → `证券卖出`
   - 其他操作全部丢弃，因此 QMT3 永远不能创建赎回篮子或现金流。
3. QMT1/QMT2 合同号转换为整数；QMT3 保留字母数字合同号。
4. QMT1/QMT2 通常没有成交时间，默认排序是：
   `交易日/交收日 → 合同号 → 来源账户 → Excel 源行号`。
5. 若共享目录存在 `QMT成交时间.csv`、`QMT1.csv` 或 `QMT2.csv`，则用：
   `来源 + 日期 + 委托号/合同号候选后缀 + 方向 + 数量`
   补真实处理时间。合同号 `3800029805` 会尝试匹配 `29805` 等后缀。
6. QMT3 排序会优先使用自身成交时间，再用合同号、账户和行号兜底。

### 5.2 QMT1/QMT2 跨账户调仓：先识别，后过账

QMT1 和 QMT2 共享一个可赎回虚拟库存，但调仓卖出必须从普通场内卖出中剥离。

第一层：逐笔保守匹配。

- 同一交易日；
- 不同来源账户；
- 一买一卖；
- 数量完全相同；
- 数字合同号间隔不超过 `transfer_contract_gap`，默认 `1000`；
- 在候选中选合同号距离最小者，再按来源和行号稳定排序；
- 支持先买后卖，也支持先卖后买。

第二层：聚合兜底。

- 对第一层未匹配记录，按 `日期 + 卖出账户 + 买入账户` 分组；
- 买卖总量必须完全相同；
- 至少一侧多笔，单买单卖不会走此兜底；
- 组内仍只配数量完全相同的单笔；
- 聚合兜底不再限制合同号间隔。

过账方式：

- 卖出按当时 FIFO 消耗旧库存；
- 调仓已实现收益 = 卖出净额 − 旧 FIFO 成本；
- 接收账户买入按自身 `abs(发生金额)` 形成新 FIFO 成本；
- 因此 QMT1/QMT2 调仓是“旧仓卖出 + 新仓买入”，不是原成本无损平移；
- 调仓已实现收益单独展示，不进入赎回篮子收益和顶部“已结算收益”。

### 5.3 QMT3 → QMT1/QMT2 成本承接

QMT3 是独立库存账，只能贡献原始成本、场内自平或向可赎回账户移仓。

识别条件：

- QMT3 卖出与 QMT1/QMT2 买入在同一交易日；
- 数量完全相同；
- 成交均价差不超过 `0.002` 元；
- 接收买入不能已被 QMT1/QMT2 调仓占用；
- 多候选时先选价格差最小，再按账户、合同号和行号排序。

当前代码事实：匹配函数没有显式校验“QMT3 卖出时间早于接收买入时间”，也不使用合同号间隔；这是后续 Agent 不应误写进代码假设的地方。

成本承接公式：

```text
接收账户新 FIFO 成本
= QMT3 原 FIFO 成本 + 接收账户买入实际成本 - QMT3 卖出净额
```

这会把两套系统之间的成交价差和费用纳入最终赎回成本。若 QMT3 原库存不足，则放弃上述承接公式，回退为接收账户买入实际成本，并保留库存不足警告。

除国内成本外，QMT3 原始开仓时的 XOP SELL 也随库存承接：

- 只使用已配置 IB 活动报表中北京时间 `09:30–15:00` 的 XOP SELL，不读取共享 `IB.csv`；
- QMT1/QMT2 视为一个虚拟系统，QMT3 视为另一个系统；调仓的接收买入不会二次创建 XOP 开仓需求；
- 每个系统按当日累计买入份额计算 `ceil(累计份额 × 990 / 1,000,000)` 的增量，避免多笔小单逐笔向上取整导致多配；
- QMT 同分钟按合同号、账户和 Excel 行号排序；IB 同分钟按活动报表源行号和稳定 ID 排序；
- 承接 XOP 分片和国内成本一起经过 QMT3→QMT1/2、QMT1↔QMT2 以及最终赎回/场内卖出；部分消耗按 FIFO 比例切片；
- 所有尚在库存中的 QMT3 关联 SELL 都从公共 IB 池按数量预留，不得被其他篮子重复使用；
- 人工 `open_trade_ids` 仍是显式覆盖：该篮子已承接的 SELL 会释放回公共池，并给出覆盖提示。

未匹配的 QMT3 卖出按普通场内自平处理。

### 5.4 FIFO、场内卖出和赎回篮子

国内账按排序后的记录逐条执行：

1. `证券买入`：向 FIFO 尾部加入库存，成本为 `abs(发生金额)`，即包含交割单实际费用。
2. `证券卖出`：先从 FIFO 头部消耗；如果已识别为调仓则进入调仓路径，否则生成 `VenueClose`。
3. `ETF 基金赎回`：从剥离场内卖出后的剩余 FIFO 中消耗，生成 `BasketResult`。

部分使用一笔买入时，数量和成本按比例拆分；所有拆分保留来源账户、日期、合同号、数量和成本，供 UI 配对图和明细审计。

库存不足不会阻止继续计算：已匹配部分仍形成暂算成本，但篮子状态会成为 `国内库存不足`，不能进入已结算合计。

### 5.5 现金差额和现金替代退款

资金流水只识别 `ETF 现金差额` 和 `ETF 申购退款`。

候选篮子必须同时满足：

- 来源账户相同；
- 合同编号相同；
- 流水日期不早于赎回日。

若合同号跨交易日重复，则按流水日期离候选篮子预计日期的距离打分：

```text
现金差额预计日 = 赎回日后第 3 个境内工作日
退款预计日     = 赎回日后第 6 个工作日

排序键 = |流水日 - 预计日|、是否早于预计日、预计日、篮子序号
```

距离相同时，预计日之后的流水优先于预计日前的流水；最后用篮子序号保证稳定结果。

同一篮子的多笔同类流水会求和；实际现金差额日/退款日取该类最后一笔流水日期。

日历口径：

- T+3 现金差额仅跳过周末，不使用 UI 维护的境外休市日；
- T+6 退款跳过周末和 `market_holidays` 手工休市日；
- 修改休市日只重算预计 T+6，不改变实际流水日期和收益。

### 5.6 人工退款

- `ib_mapping_overrides.json` 可保存 `manual_refund_amount`；
- 只有金额大于 0 且 QMT 尚无实际 `ETF 申购退款` 时才应用；
- 人工退款可以让已满足其他条件的篮子变为 `已结算`；
- 一旦 QMT 实际退款出现，交割单金额自动优先，人工值保留但不参与计算。

## 6. IB 端匹配原则与优先级

### 6.1 IB 输入

从 IB 活动报表读取：

- `交易,Data` 中证券代码为 `XOP` 的成交；
- 所有资产类别为“股票”的成交，用于识别成分券同步做空窗口；
- `借入费用详情,Data` 中 XOP 的借券费。

IB 交易按 `时间 → 源行号 → 稳定交易 ID` 排序。稳定 ID 由原始字段和同签名出现次数生成，使人工映射可跨重算保存。

正式主账的 IB 输入只是 `config.json.ib_path` 指向的活动报表。共享目录当前仅用于 QMT 时间提示和实时 JSON，共享 `IB.csv` 不在读取链路中。

### 6.2 直接空头、XOP 底仓和合成开仓

程序不会简单把 IB 的 `O/C` 标记当作策略意图，而是按成交数量和成分券窗口重建 XOP 等价流。

成分券同步做空窗口：

- 同一天至少出现 `10` 个不同股票代码的 SELL；
- 窗口从当天最早相关 SELL 前 `5` 分钟，到最晚相关 SELL 后 `20` 分钟。

XOP 处理顺序：

1. XOP SELL 先卖已有 XOP 底仓，使用部分生成 `synthetic_open`；
2. SELL 剩余部分生成 `direct_open`，增加直接空头；
3. XOP BUY 先回补已有直接空头，生成 `direct_close`；
4. BUY 剩余部分只有位于成分券窗口时才进入 XOP 底仓；
5. 默认情况下仍未解释的 BUY 不进入正式等价流；虚拟平仓重算时可保留为 `direct_close` 候选。

合成空头暂按卖出 XOP 的价格计价，不逐只成分券计算成本；XOP 底仓买入本身不直接作为赎回篮子的回补。

### 6.3 人工映射预留

人工为某篮子保存的 `open_trade_ids` / `close_trade_ids` 会先从自动池预留，避免被其他篮子或场内自平抢占。

- 人工开仓不受默认“赎回日美东 15:00 前”的限制；
- 人工平仓从所选开仓最早时间之后开始；
- 一个原始成交拆出的派生 ID，可用完整 ID 或冒号前的原始 ID 选中；
- 启用人工虚拟平仓时会清除人工 `close_trade_ids`，防止真实 BUY 与虚拟 BUY 双重平仓。

### 6.4 分配优先级：场内自平先于赎回篮子

IB 分配是全局容量分配，同一股成交只能使用一次。

第一步：识别国内场内自平对应的 XOP BUY。

- 将 IB 时间从美东转换为北京时间；
- 只取国内场内卖出当天北京时间 `09:30–15:00` 的 XOP BUY；
- 有 QMT 实际成交时间时，按时间距离最近排序，同距离时优先卖出后的 BUY；
- 没有实际时间时按北京时间、价格、行号稳定排序；
- 先占用这些 BUY，匹配数量仍按 `场内卖出份额 × 990 / 1,000,000` 向上取整。

第二步：建立时间事件。

- 场内自平事件时点 = 已匹配 BUY 中最晚时间；
- 赎回篮子事件时点 = 赎回日美东 `15:00`；
- 所有事件按时间排序，同一时点场内自平先于篮子。

第三步：分配 SELL 开仓。

- 场内自平从事件时点以前的 SELL 中取开仓；
- 自动篮子从赎回日美东 15:00 以前的 SELL 中取开仓；
- 按全局成交顺序 FIFO 消耗容量。

第四步：分配篮子 BUY 平仓。

- 自动篮子只取赎回日美东 15:00 及之后的 BUY；
- 按成交顺序 FIFO 消耗；
- 先前已被场内自平占用的 BUY 不再可用。

### 6.5 IB 成交切片

若一笔 IB 成交只使用一部分：

```text
使用比例 = 采用股数 / 原成交绝对股数
分片成交额 = 原成交额 × 使用比例
分片佣金   = 原佣金 × 使用比例
```

未使用容量保留给后续事件；最终仍未使用的 SELL/BUY 分别计入 `unallocated_ib_sell_qty` 和 `unallocated_ib_buy_qty`，不会强行塞入篮子。

### 6.6 次交易日国内买入虚拟平仓

当篮子没有足量真实 XOP BUY，程序可用下一境内工作日未被场内平仓/调仓消耗的 `159518` 买入生成一对 XOP 等价成交：

- `domestic_rollover_close`：为旧篮子平仓；
- `domestic_rollover_open`：以相同价格、相同数量立即为新国内持仓重开空头。

换算方式：

```text
国内买入可代表的 XOP 股数 = 国内份额 × 990 / 1,000,000，向上取整
虚拟成交额 USD = abs(国内买入发生金额) × 采用比例 / 主界面人工汇率
虚拟 XOP 价格 = 虚拟成交额 USD / XOP 股数
```

自动触发条件：真实平仓不足一半；人工 `manual_virtual_close=true` 时强制尝试。允许最终缺口不超过目标的约 `5%`（`max(1, target // 20)`），否则整组候选回滚、不生成虚拟成交。

当前“下一交易日”只跳过周末，不读取手工休市日。

### 6.7 借券费

借券费分摊给费用起息日前仍未平仓的 `direct_open` 和 `qmt3_carried_open`：

```text
某篮子活动直接空头
= 起息日前（direct_open + qmt3_carried_open）数量 - 起息日前所有 close 数量

某篮子借券费
= 该费用金额 × 篮子活动直接空头 / 全部篮子活动直接空头
```

`synthetic_open` 不参与借券费分摊。当前实现只给赎回篮子分摊，不给 `VenueClose` 分摊；IB 费用行自带的 `qty` 字段被读取但不参与比例公式。

## 7. 收益计算原则

### 7.1 正式赎回篮子

国内部分：

```text
国内成本 = 赎回消耗的 FIFO 买入成本之和
国内收益 = ETF申购退款 + ETF现金差额 - 国内成本
```

其中退款可来自 QMT 实际流水或当前生效的人工退款；现金差额保留正负号。

IB 交易部分：

```text
IB交易收益 USD
= SELL分片成交额 - SELL分片佣金
 - BUY分片成交额 - BUY分片佣金

IB净收益 USD = IB交易收益 USD - 分摊借券费 USD
IB收益 RMB   = IB净收益 USD × 主界面人工汇率
篮子合计 RMB = 国内收益 + IB收益 RMB
```

主账使用同一个 UI 人工汇率换算所有历史篮子和场内自平，不按日期自动取 CFETS。

内部尽量使用 `Decimal` 保留精度；金额展示和 CSV 金额字段通常按人民币分四舍五入，IB 细项可保留 6 位。

### 7.2 场内自平

```text
国内场内收益 = 卖出净额 - 国内 FIFO 成本
IB场内收益USD = 对应SELL成交额 - SELL佣金 - 对应BUY成交额 - BUY佣金
IB场内收益RMB = IB场内收益USD × 主界面人工汇率
场内合计RMB   = 国内场内收益 + IB场内收益RMB
```

场内自平单独展示，不进入赎回篮子和顶部“已结算收益”。当前场内路径不分摊借券费。

### 7.3 调仓

```text
调仓已实现收益 = 卖出净额 - 卖出消耗的旧 FIFO 成本
```

QMT1/QMT2 接收仓以实际买入成本重新建仓；QMT3 成本承接使用第 5.3 节的跨系统成本公式。调仓收益单独展示，不进入顶部已结算篮子合计。

### 7.4 顶部已结算合计

```text
顶部“已结算收益”
= 所有 status == "已结算" 的赎回篮子 total_pnl_cny 之和
```

不包含：

- 未结算篮子的暂算收益；
- 场内自平收益；
- 跨账户调仓已实现收益。

## 8. 结算状态判定

状态按以下优先级确定，前项命中后不再看后项：

1. `国内库存不足`：赎回 FIFO 有缺口；
2. `IB未完全匹配`：IB 开仓或平仓有缺口；
3. `待现金差额`：没有实际 `ETF 现金差额`；
4. `待现金替代款`：已有现金差额，但既无 QMT 实际退款，也无当前生效人工退款；
5. `已结算`：以上全部完整。

未结算篮子仍显示暂算收益，但不会进入顶部已结算合计。

## 9. 预测、校准和实时估值公式

### 9.1 PCF 隐含 XOP 股数

PCF 中 `NAVperCU`、`CashComponent` 对应 `PreTradingDay` 的估值口径：

```text
q_nav = NAVperCU / SAFE中间价 / XOP估值日收盘价
q_net = (NAVperCU - CashComponent) / SAFE中间价 / XOP估值日收盘价
```

默认 `chosen_q = q_net`。

- `q_net` 超出 `980–1010`：警告，置信度低；
- `q_nav - q_net` 绝对值大于 2 股：提示现金差额影响较大；
- `q_net` 在 `990–1000` 且无警告：高置信度；
- `980–1010` 内但有其他情况：中置信度。

### 9.2 指定日期/QMT 篮子到账估算

```text
单位比例 = 赎回份额 / 最小申赎单位
预估XOP股数 = 采用的动态每单位股数 × 单位比例
预估退款 = 预估XOP股数 × XOP卖出价格窗口 × 赎回日CFETS收盘价
预估现金差额 = PCF EstimateCashComponent × 单位比例
预估总回款 = 预估退款 + 预估现金差额
预估国内收益 = 预估总回款 - 国内FIFO成本
```

对已有 QMT 篮子，若实际现金差额已经出现，则用实际值替代 PCF 预估值；实际退款只用于对照，不覆盖预测退款。

实际到账反推：

```text
实际反推每单位股数
= 实际ETF申购退款 / CFETS收盘价 / XOP价格代理 / 单位比例
```

只有 `980–1010` 的反推结果标记为可纳入校准样本。现金差额不进入股数反推，只参与实际总回款。

### 9.3 篮子资产预测缓存

用于篮子汇总“预估篮子资产”，固定采用：

```text
预测股票退款 = 996 × 单位比例 × XOP美东15:59一分钟收盘价 × T日CFETS 16:00
预测现金差额 = T日PCF EstimateCashComponent × 单位比例
预测篮子资产 = 预测股票退款 + 预测现金差额
预估偏差 = 实际篮子资产 / 预测篮子资产 - 1
```

只有实际退款和实际现金差额都到齐时才显示偏差。缓存必须匹配篮子 ID、日期、合同号、份额和模型版本 `total_asset_996_1559_cfets1600_v1`；旧模型缓存不直接复用。

### 9.4 实时溢价率

```text
证券资产Bid = 996 × XOP Bid × CFETS最新可用小时价
证券资产Ask = 996 × XOP Ask × CFETS最新可用小时价
总篮子Bid = 证券资产Bid + 当日PCF EstimateCashComponent
总篮子Ask = 证券资产Ask + 当日PCF EstimateCashComponent
Bid净值 = 总篮子Bid / 1,000,000
Ask净值 = 总篮子Ask / 1,000,000

同侧买一溢价 = 159518买一 / Bid净值 - 1
同侧卖一溢价 = 159518卖一 / Ask净值 - 1
保守卖国内 = 159518买一 / Ask净值 - 1
保守买国内 = 159518卖一 / Bid净值 - 1
```

实时页要求 XOP Bid/Ask、159518 买一/卖一、CFETS 和同日 PCF 现金差额全部存在。PCF 缺失时停止估值，绝不把现金差额默认为 0。

CFETS 实时价取当天最新小时字段；当天没有则向前最多查 10 天的最近工作日。共享 `realtime.json` 使用 schema 2 原子覆盖写入。

## 10. 特殊情况处理清单

| 特殊情况 | 当前处理 |
|---|---|
| QMT2 未配置 | 按空账户处理并给全局提示 |
| QMT3 未配置 | 按空独立成本账户处理 |
| QMT3 出现赎回/现金流 | 读取阶段直接忽略 |
| QMT3 开仓 XOP 不足 | 仅关联中国盘中已有 SELL，记录溯源缺口，不使用盘后卖出补齐 |
| QMT3 关联 SELL 尚未赎回 | 仍按数量锁定在原国内库存，不计入未分配 SELL |
| QMT1/QMT2 无成交时间 | 用合同号顺序作为日内代理；可用共享 CSV 补时间 |
| QMT 合同号跨日重复 | 现金流在同账户同合同候选中按 T+3/T+6 距离二次归属 |
| 一笔国内/IB 成交被多处使用 | 按实际数量和成本/佣金比例切片，不整笔绑定单一篮子 |
| 国内库存不足 | 保留已匹配暂算，状态优先标红并排除已结算合计 |
| IB 开仓或平仓不足 | 保留暂算和缺口，状态为 `IB未完全匹配` |
| IB 多余成交 | 留在未分配 SELL/BUY，不强行归属 |
| 中国盘中场内卖出对应 IB BUY | 在赎回篮子之前剥离，避免被后续篮子误吃 |
| 成分券同步卖出附近有 XOP BUY | 先视作底仓；日后卖出底仓形成 `synthetic_open` |
| 没有真实 XOP BUY 平仓 | 可自动或人工使用次日国内买入生成虚拟 close/open |
| 人工选择 IB 成交 | 所选 ID 先全局预留，并持久化到 overrides |
| 人工虚拟平仓与人工真实平仓冲突 | 开启虚拟平仓时清除 `close_trade_ids` |
| 人工退款后 QMT 实际退款到账 | 自动改用 QMT 实际值 |
| 美国休市日影响 T+6 但不应影响境内 T+3 | T+6 使用手工休市日，T+3 只跳周末 |
| XOP 指定价格窗口缺失 | `market_data.py` 按窗口定义回退到其他收盘字段，最终可回退日收盘 |
| 预测要求 15:59 价格但缺失 | 预测重算明确报错，不用普通日收盘冒充 |
| 实时 PCF 缺失 | 停止总资产估值，不按 0 计算 |
| 新浪集合竞价只有一档 | 共享 JSON 的买二/卖二复用买一/卖一，保持四个数值档位 |
| SSE 历史 PCF 不可直接下载 | 不用最新 PCF 回填历史日期；缓存交易日不符则拒绝 |
| PCF 请求触发 403/429/503 | 持久化冷却期；冷却期内不重复请求 |
| PCF 连续瞬时失败 | 第二次起进入短冷却 |
| 已配置的 QMT/IB 数据文件被覆盖更新 | 文件监听器短延迟触发重算并重新挂载监听 |
| 条件单是周末、已过期、数量错误或 orderRef 重复 | 禁止发送 |
| TWS 断开时队列仍有已确认订单 | 清空队列并逐笔发出 rejected 事件 |

## 13. 当前实现中的重要事实与潜在改动风险

1. `redemption_engine.d()` 对空值或无法解析的数字返回 `0`，部分坏行会被跳过而不是立刻失败。若改为严格校验，会改变现有容错和输入兼容性。
2. 篮子 ID 由 `来源 + 日期 + 合同号 + 份额` 哈希生成。若同账户同日出现完全相同合同号和份额的重复赎回，两条记录会得到相同 ID；人工覆盖和预测缓存可能冲突。
3. 主界面历史计算使用一个人工汇率处理所有篮子，不是逐日 CFETS。不要因预测模块已接入 CFETS 就默认主账也是逐日汇率。
4. QMT1/QMT2 聚合调仓兜底会忽略合同号间隔，但仍要求组内单笔数量可以一一相等匹配。
5. QMT3 成本承接当前不检查卖出与买入的时间先后，只检查同日、精确数量和价格容差。
6. 借券费行的 `qty` 当前未用于分摊；分母来自各篮子的活动 `direct_open + qmt3_carried_open`。
7. 场内自平和调仓收益不计入顶部“已结算收益”，该顶部数字不是账户全口径总收益。
8. 主引擎公开参数支持 `hedge_targets` 动态覆盖，但当前 UI 主计算没有传入该参数，因此正常桌面使用仍固定 990。
9. 自动虚拟平仓的“下一交易日”只跳周末，不跳人工休市日。
10. `159605` 是全现金替代研究，不应进入 `159518` 的 XOP 实物等价主账模型。
